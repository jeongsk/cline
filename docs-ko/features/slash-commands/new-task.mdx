`/newtask`는 완벽한 개발자 인수인계처럼 작동하는 슬래시 명령어입니다. 전체 계획, 완료된 작업, 관련 파일 및 다음 단계 등 중요한 사항을 지능적으로 패키징하여 깨끗한 컨텍스트 창이 있는 새 작업으로 만듭니다. 동시에 도구 호출, 문서 검색 및 구현 세부 정보의 노이즈는 남겨둡니다.

이는 새 개발자를 프로젝트에 투입할 때 수행하는 것과 정확히 동일합니다. 이전의 모든 키 입력을 부담스럽게 하지 않고 작업을 계속하는 데 필요한 필수 컨텍스트를 제공합니다.

#### `/newtask` 슬래시 명령어 사용

컨텍스트 창이 가득 찼지만 프로젝트가 아직 완료되지 않은 경우:

-   채팅 입력 필드에 `/newtask`를 입력합니다.
-   Cline이 대화를 분석하고 전달할 컨텍스트의 요약 버전을 제안합니다.
-   커밋하기 전에 대화를 통해 이 제안된 컨텍스트를 구체화할 수 있습니다.
-   만족스러우면 구체화된 컨텍스트로 새 작업을 만드는 버튼이 나타납니다.

#### 예시

여러 단계로 구성된 복잡한 구현 작업을 할 때 `/newtask`를 정기적으로 사용합니다. 예를 들어 10단계 프로세스 중 3단계를 완료했고 컨텍스트가 이미 문서 스니펫, 파일 내용 및 상세한 토론으로 75% 가득 찼다고 가정해 보겠습니다.

이러한 통찰력을 잃거나 처음부터 다시 시작하는 대신 `/newtask`를 사용하여 개별 도구 호출 및 조사 단계의 모든 노이즈 없이 주요 결정, 파일 변경 사항 및 현재까지의 진행 상황 등 중요한 사항을 Cline이 추출하도록 합니다.

저는 `/newtask`를 프로젝트에 새로 합류한 개발자라고 생각하고 싶습니다. 완료된 작업에 대한 완전한 이해, 관련 파일에 대한 인식, 도움이 될 만한 기타 컨텍스트 및 다음에 수행할 작업에 대한 정보를 제공해야 합니다.

#### 영감

다음은 `/newtask`를 사용하는 몇 가지 인기 있는 방법입니다.

-   Context7 MCP 서버를 사용하여 복잡한 API를 조사하고 컨텍스트를 문서로 채웁니다. 개념을 이해한 후 `/newtask`를 사용하여 구현에 필요한 필수 지식만으로 새로 시작합니다.
-   여러 번의 디버깅 시도와 파일 탐색을 통해 어려운 버그의 근본 원인을 파악한 후 `/newtask`를 사용하여 실패한 모든 시도를 버리고 솔루션만 포함된 깨끗한 상태에서 계속합니다.
-   클라이언트 토론에서 여러 접근 방식을 탐색하고 마침내 한 방향으로 결정되면 `/newtask`를 사용하여 선택한 솔루션 구현에만 집중합니다.
-   여러 날에 걸쳐 진행되는 복잡한 프로젝트의 경우 논리적 중단 지점에서 `/newtask`를 사용하여 진행 상황을 전달하면서 깨끗한 작업 공간을 유지합니다.
